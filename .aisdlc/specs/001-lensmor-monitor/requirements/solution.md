## 1. 结论摘要

- **一句话目标**：为产品经理、市场负责人、创始人、投资人提供一个自动化竞品情报监控 Web 平台，将手动竞品调研变为定期收取结构化情报卡片。
- **本次 In / Out**：In = 核心情报闭环（引导→监控→采集→分析→收件箱→反馈）；Out = 历史差异分析(90天)、战略总结、社媒分析、公司基本面自动提取。
- **推荐方案**：Web Dashboard 单体应用 + 异步任务队列，前端 SPA + 后端 API + 后台 Worker，分阶段交付。
- **优先验证点**：V-001（竞品发现数据源可行性）、V-002（网站差异分析质量阈值）、V-003（采集性能瓶颈）。

## 2. 推荐方案：Web Dashboard + Async Worker

- **方案名**：Dashboard-First Monolith with Async Pipeline
- **主流程 / 关键机制**：
  1. 用户注册/登录后进入 3 步引导（角色选择→自有产品信息→导入竞品）
  2. 竞品管理左侧边栏，状态实时展示（监控中/已暂停/采集进行中）
  3. 后台 Worker 按配置间隔定时采集竞品网站，经数据清洗管道（HTML→MD→去噪→摘要→差异标注）
  4. AI 分析管道产出网站差异分析报告，推送到用户收件箱
  5. 收件箱（左侧卡片列表+右侧详情），支持筛选、标记已读、提交反馈
  6. 反馈数据回灌分析管道，持续改进质量
- **关键边界/取舍**：
  - MVP 不做历史差异分析（90 天串联）和战略总结——先交付"单次差异分析"闭环
  - 竞品推荐依赖外部数据源（如 SimilarWeb/Crunchbase API 或自建竞品库），MVP 可能降级为"手动添加 + 有限自动建议"
  - 数据采集通道（4.1）详情待补充，MVP 先实现网站 HTML 采集通道，其余通道后续迭代
  - 用户认证采用标准 JWT + 邮箱密码登录，不做 SSO
  - 不实现实时推送通知，用户主动登录查看收件箱
- **为什么选它**：
  - MVP 范围（核心情报闭环）已由用户确认，与方案一一对应
  - 单体架构降低初期基础设施复杂度，后续可按 Worker 负载拆分
  - 先验证"网站差异分析"核心价值假设，再扩展分析类型

## 3. 备选方案

### 3.1 备选方案：No-Code 集成优先

- **核心机制**：最小化自研代码，用第三方工具链（Zapier/Make + 现成爬虫服务 + ChatGPT API + Notion/飞书多维表格作为"收件箱"）拼装 MVP。
- **主流程**：用户填写 Google Form 配置竞品 → 定时触发爬虫 → ChatGPT 分析差异 → 结果写入 Notion/飞书 Base → 用户查看。
- **边界与取舍**：零前端开发、零部署；但定制化能力极低，无法支持引导流程、角色选择、反馈闭环。
- **适用前提**：只做 1-2 周内的概念验证，团队无前端/后端开发资源。
- **不选原因**：用户已明确期望"Web 平台"形态，需要完整的引导流程、竞品管理 UI、收件箱交互——No-Code 方案无法满足。

### 3.2 备选方案：API-First 微服务

- **核心机制**：先建完整 API 层（竞品管理 API、采集调度 API、分析 API、收件箱 API），前端作为薄展示层；采集和分析各自独立微服务。
- **主流程**：与推荐方案相同，但每个管道阶段（采集/清洗/分析/报告）独立部署、独立扩展。
- **边界与取舍**：扩展性好，但 MVP 阶段微服务基础设施（服务发现、配置中心、CI/CD）成本远高于收益。
- **适用前提**：团队已有 Kubernetes 集群和微服务治理经验，且首版就预期高并发采集。
- **不选原因**：MVP 阶段无需微服务复杂度；单体 + 异步 Worker 已足够，后续可按采集负载拆分。

## 5. 验证清单（V-xxx，可执行）

- V-001：竞品发现数据源可行性
  - 风险/假设：竞品自动推荐依赖外部数据源（如 SimilarWeb/Crunchbase API 或自建竞品库），MVP 若不可用需降级
  - 方法：调研至少 2 个潜在数据源，评估 API 可用性、覆盖度、成本
  - 成功/失败信号：找到 ≥1 个可用的竞品发现数据源（覆盖 ≥50% 常见竞品）为成立
  - Owner：DEV
  - 截止：设计阶段结束前
  - 触发动作：不成立 → MVP 降级为"手动添加竞品 + 基于 URL 的基础元数据抓取"，竞品推荐移至 V2

- V-002：网站差异分析质量阈值
  - 风险/假设：AI 网站差异分析的质量直接影响核心价值——假阳性/假阴性会损害用户信任
  - 方法：选取 5 个已知有变更的竞品网站和 5 个无变更的网站，跑差异分析管道，人工判定准确性
  - 成功/失败信号：准确率 ≥80%（真正例+真负例/总数）且假阳性率 ≤20%
  - Owner：DEV
  - 截止：实现阶段中期（采集管道完成后 3 天内）
  - 触发动作：不成立 → 调整差异阈值/引入人工审核环节/增加数据清洗步骤

- V-003：采集性能瓶颈识别
  - 风险/假设：竞品网站采集可能因频率限制、页面大小、JS 渲染导致超时或 IP 封禁
  - 方法：对 10 个不同类型的竞品网站执行采集，记录成功率、耗时、被封情况
  - 成功/失败信号：单次采集 p95 < 30s 且成功率 ≥90%
  - Owner：DEV
  - 截止：实现阶段早期（首个采集 Worker 完成后）
  - 触发动作：不成立 → 引入代理池/调整采集频率/JS 渲染降级为纯 HTML

- V-004：数据采集通道（4.1）补齐
  - 风险/假设：4.1 节"暂无法展示"的采集通道详情可能包含对架构有重大影响的约束（如特定协议、数据源限制）
  - 方法：从飞书文档中获取 4.1 完整内容，评估对方案的影响
  - 成功/失败信号：4.1 内容已获取且不影响推荐方案核心架构
  - Owner：PM
  - 截止：设计阶段开始前
  - 触发动作：若含重大架构约束 → 重新评估推荐方案并更新本文

- V-005：引导流程角色与情报解读关联
  - 风险/假设：角色选择影响"情报解读角度"这一需求若实现复杂（需要不同 prompt 或分析逻辑），可能超出 MVP 范围
  - 方法：设计阶段评估角色差异化解读的实现成本（prompt 参数化 vs 独立分析逻辑）
  - 成功/失败信号：可通过 prompt 参数化实现（无需独立分析管道）
  - Owner：DEV
  - 截止：设计阶段结束前
  - 触发动作：若需独立分析管道 → 标记为高成本项，考虑 MVP 降级为"角色仅影响默认筛选视图"

## 8. Mini-PRD

（基本版——用于支撑后续 routing 判定是否需要完整 PRD）

- **MVP 范围**：
  - In：3 步引导、竞品管理(CRUD/暂停/手动刷新)、竞品详情 Overview、分析报告收件箱(含详情)、情报反馈(有用/有误/不重要)、网站差异分析管道、数据清洗管道、异步任务队列
  - Out：历史差异分析(90天)、战略总结、社媒品牌分析、公司基本面自动提取、实时推送通知
  - Known Gap：数据采集通道(4.1)详情待补充
- **验收标准（AC）**：
  - AC1：新用户可完成 3 步引导（角色选择→产品信息→竞品导入）
  - AC2：用户可添加/编辑/删除/暂停/恢复竞品，刷新后可查看采集状态
  - AC3：点击竞品可查看 Overview（公司基本面 + 最新情报时间线）
  - AC4：收件箱左侧卡片列表展示分析报告，右侧展示详情（变更摘要、战略意图、行动建议）
  - AC5：收件箱支持按优先级/日期/竞品筛选，打开即标记已读
  - AC6：用户可对情报提交反馈（有用/有误/不重要），有误时可选错误类型
  - AC7：后台定时任务可自动采集竞品网站，触发差异分析并生成报告
  - AC8：用户操作需登录认证，只能访问自己名下的数据
- **交互变化结论**：全新产品，所有页面均为新建。涉及页面：引导流程(3 步)、竞品管理(左侧边栏+右侧详情)、收件箱(卡片列表+详情) —— 建议进入原型阶段消除交互歧义。
- **影响面**：全新项目，所有模块从零构建。前端 SPA + 后端 API + Worker + DB。
